## 驗證紀錄：LWW 衝突解決（Issue #6）

針對 Issue #6 的 `LWW (Last-Write-Wins)` 衝突合併機制，我們編寫了 `scratch_lww_test.ts` 實驗腳本以模擬 Fiddler 封包行為攔截請求，確保當終端設備更新了過期舊資料並送出時，後端可成功阻擋。

### 測試步驟與行為
1. **登入驗證**：呼叫 `POST /api/v1/auth/login` 利用 `admin_test` 與已知密碼 `P@ssw0rd!` 登入取得 JWT Token。
2. **Push 模擬新增 (`operationType: create`)**：新增一個全新的 `Customer` ('LWW 測試公司')。
3. **Pull 拉取最新伺服器狀態**：從資料庫拉取剛建立成功之實體，獲得伺服器實際的 `id` 值與 `updatedAt`（例如 `09:52:29`）。
4. **Push 模擬過期覆蓋 (`operationType: update`)**：修改 `updatedAt` 加以竄改，使其倒退一小時（例如 `08:52:29`），試圖以此過期快照覆蓋伺服器。

### 預期與實際結果：✅ 驗證成功
後端攔截機制精準捕獲了過期 `updatedAt` 比較 (`current.updatedAt > payloadUpdatedAt`)，並以嚴格吻合同步協定 v1.6 (§6 錯誤碼) 的格式擋下操作：
```json
{
  "succeeded": [],
  "failed": [
    {
      "operationId": "7d44e556-173b-4433-8c27-a99450570053",
      "code": "FORBIDDEN_OPERATION",
      "message": "LWW：伺服器資料較新，執行 Force Overwrite。",
      "server_state": {
        "entityType": "customer",
        "id": 2,
        "name": "LWW 測試公司 1775987549337",
        "updatedAt": "2026-04-12T09:52:29.353Z"
      }
    }
  ]
}
```
**結論**：LWW 衝突解決防護網確認生效。未來當 App 前端收到 `FORBIDDEN_OPERATION` 並附帶 `server_state` 時，其內建的 `clearOrphanedOfflineCustomers` 與 `upsertCustomerFromServer` 將會自動放棄自身並妥善覆寫成伺服器最新紀錄。
