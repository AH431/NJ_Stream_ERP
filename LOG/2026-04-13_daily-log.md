### 雙重保護設計

```
應用層（先）：
  if (newOnHand < 0 || newReserved < 0 || newReserved > newOnHand)
    → INSUFFICIENT_STOCK + server_state（格式統一，前端可直接處理）

DB 層（兜底）：
  CHECK (quantity_on_hand >= 0)
  CHECK (quantity_reserved <= quantity_on_hand)
```

**刻意不依賴 DB CHECK 作為主要錯誤回應**：DB CHECK 觸發時只拋出 PG 原生錯誤，難以包裝成同步協定 v1.6 §6 標準格式。應用層先攔，DB CHECK 只作安全兜底。

## 併發衝突模擬測試 — Race Condition 完整驗收 ✅

**涵蓋範圍**：Issue #11 + 協定補強（INSUFFICIENT_STOCK → HTTP 409）

### 場景 1：庫存 Race — INSUFFICIENT_STOCK 409 + Force Pull

| 步驟 | 操作 | 結果 |
|------|------|:----:|
| 前置 | productId=3, onHand=12, reserved=0 | — |
| 裝置 A | reserve 8 | HTTP 200 succeeded，reserved: 0→8 |
| 裝置 B | reserve 5（8+5=13 > 12）| HTTP **409**，INSUFFICIENT_STOCK |
| 裝置 B Force Pull | GET /pull | quantityOnHand=12, quantityReserved=8 ✅ |

**結論**：後端回 409（非 200），前端 DioException 攔截 409 → 自動 pullData()，庫存快照與伺服器一致。

### 場景 2：LWW Race — 客戶更新衝突 + Force Overwrite

| 步驟 | 操作 | 結果 |
|------|------|:----:|
| 前置 | customer id=2, server.updatedAt = T_s | — |
| 裝置 A | update name, payloadUpdatedAt=T_s（等值→接受）| HTTP 200 succeeded，server.updatedAt → T_a（now）|
| 裝置 B | update name, payloadUpdatedAt=T_s（< T_a）| HTTP 200，**FORBIDDEN_OPERATION** |
| server_state | 含 A 版 name + 最新 updatedAt | 前端 Force Overwrite 覆蓋本地 B 版 ✅ |

**結論**：LWW 嚴格生效（`current.updatedAt > payloadUpdatedAt`），後同步裝置收到 server_state 直接 Force Overwrite，不需手動 Pull。

### 場景 3：First-to-Sync wins — 報價轉訂單競爭

| 步驟 | 操作 | 結果 |
|------|------|:----:|
| 前置 | quotation id=2, convertedToOrderId=null | — |
| 裝置 A | sales_order:create（quotationId=2）| HTTP 200 succeeded，quotation.convertedToOrderId → 3 |
| 裝置 B | 同一 quotation id=2 再轉單 | HTTP 200，**FORBIDDEN_OPERATION** |
| server_state | quotation.convertedToOrderId=3, status="converted" | 前端 Force Overwrite 後知報價已被轉換 ✅ |
| 裝置 B Force Pull | quotation.convertedToOrderId=3 | 確認最終一致 ✅ |

**結論**：同一報價只有第一個到達伺服器的裝置能成功轉換，DB transaction 原子性保護不會雙重建單。
