以下是針對用 **Llama 3B 小模型**（如 Llama 3.2 3B Instruct）跑企業內部資料（客戶資料、報價單、產品資訊、物料查詢等）的 **知識卡片（Knowledge Card / Chunk）規劃建議**。目標是「**最大化效益**（檢索準確率高、回答可靠、幻覺少） + **最小化系統負擔**（向量資料庫小、檢索快、context 占用少、小模型容易處理）」。

### 1. 核心原則（小模型 + 企業結構化/半結構化資料的關鍵）
Llama 3B 上下文理解和複雜推理能力有限（相較 8B/70B），因此**不要餵大段文字**，要讓每個「知識卡片」盡量**自包含、單一焦點、易理解**。

- **單一主題原則**：一個卡片只解決「一件事」或「一個實體」，避免混合多個客戶/產品。
- **自包含原則**：卡片內要包含足夠上下文，讓小模型不用跨多張卡片就能回答（但不要過長）。
- **明確結構原則**：用清晰的標題、欄位名稱、Key-Value 格式，讓 embedding 和生成都更容易抓住重點。
- **粒度平衡**：太細 → 檢索碎片化、需要多張卡片拼湊（小模型容易混亂）；太粗 → 向量稀疏、檢索不準、context 浪費。
- **推薦 Chunk 大小**：**300–600 tokens**（約 200–400 字中文），overlap 10–20%。這對 3B 模型和大多數 embedding 模型都很友好。

### 2. 不同資料類型的知識卡片設計建議

#### **A. 客戶資料（CRM）**
- **推薦方式**：**以「客戶 ID / 客戶名稱」為單位做一張或少數幾張卡片**。
- 內容結構範例（Markdown 或純文字）：
  ```
  客戶：XYZ 科技股份有限公司 (客戶ID: CUST-2025001)
  類型：企業客戶
  聯絡人：王經理 (電話: 02-XXXXXXX, Email: wang@xyz.com)
  主要需求：工業自動化設備、客製化報價
  歷史報價摘要：2025 Q1 報價單 #Q250301 (總額 NT$1,850,000，已成交)
  目前狀態：合作中，信用額度良好，最近互動 2026-04-10
  注意事項：偏好 30 天付款條件，對交期敏感
  ```
- **好處**：查「王經理的客戶」或「XYZ 的歷史報價」時，一張卡片就能涵蓋大部分資訊，減少小模型的拼湊壓力。
- 如果客戶資料非常多，可再細分成「基本資料卡 + 互動歷史卡」，但盡量控制在 2 張以內。

#### **B. 報價單**
- **推薦方式**：**一張報價單做 1–2 張卡片**（不要整份 PDF 直接切碎）。
  - 第一張：報價單摘要 + 總覽（客戶、日期、總金額、狀態）。
  - 第二張（如果品項多）：重要品項列表（用表格或清單，控制在 5–10 項重點）。
- 結構範例：
  ```
  報價單：Q250301 - XYZ 科技 (日期: 2025-03-15，有效期至 2025-04-15)
  客戶：XYZ 科技股份有限公司
  總金額：NT$1,850,000 (稅後)
  狀態：已成交
  付款條件：30 天
  主要品項：
  - 工業控制器 (型號: IC-8800) × 50 台，單價 NT$12,500，小計 NT$625,000
  - 感測器模組 (型號: SEN-320) × 200 組，單價 NT$2,800，小計 NT$560,000
  ...
  備註：含安裝指導，交期 25 天
  ```
- **提示**：報價單常有表格，建議先用工具（如 LlamaIndex/Unstructured 或程式）把表格轉成 Markdown 清單或 Key-Value，避免 embedding 對表格理解差。

#### **C. 產品 / 物料查詢（最常見的高頻查詢）**
這部分最需要精細設計，因為查詢多樣（型號、規格、相容性、價格、庫存等）。

- **最佳做法**：**以「單一產品/物料」為單位**，做一張結構化知識卡片。
- 結構範例（強烈建議用這種半結構化格式）：
  ```
  產品/物料：工業控制器 IC-8800
  類別：自動化控制系統
  關鍵規格：
  - 輸入電壓：DC 24V
  - 輸出通道：16 路繼電器
  - 通訊介面：Modbus RTU / Ethernet
  - 工作溫度：-10°C ~ 60°C
  - 尺寸/重量：150x100x50mm / 450g
  價格參考：單價 NT$12,500 (2026 年版，數量折扣適用)
  相容產品：SEN-320 感測器、ACT-4500 致動器
  庫存狀態：充足 (2026-04 更新)
  注意事項：不適合高濕度環境，建議搭配穩定電源
  應用場景：生產線自動化、設備監控
  ```
- **為什麼這樣好？**
  - Embedding 時容易捕捉「IC-8800 + 規格」等關鍵語義。
  - 小模型生成時，資訊條理清楚，不易幻覺。
  - 支援跨屬性查詢（如「找支援 Modbus 的控制器」）。

- 如果產品屬性非常多（20+ 欄位），可以考慮**表格知識庫模式**（很多 RAG 框架如 Dify/FastGPT 支援直接匯入 Excel，按欄位檢索），或把「核心規格」和「進階/應用」拆成兩張卡片。

#### **D. 其他輔助卡片**
- **常見問題 FAQ 卡**：把重複詢問的問題（如「報價有效期多久？」「物料退貨流程？」）做成獨立卡片，一問一答格式，準確率最高。
- **術語/詞彙表卡**：公司內部專有名詞、單位換算、物料編碼規則等，單獨成卡。
- **關係卡**（可選）：產品與客戶偏好、物料與產品的對應關係，用簡單句子描述。

### 3. 整體規劃與優化技巧（最小化負擔）
1. **前處理流程**：
   - 原始資料（Excel、PDF、資料庫）→ 結構化清洗 → 轉成 Markdown/JSON-like 文字 → Chunking。
   - 使用 **Recursive Character Text Splitter** + 語義感知分割（semantic chunking 如果資源允許），優先保留表格和清單的完整性。

2. **Embedding 模型選擇**（很重要！）：
   - 推薦輕量高效的多語言模型：**BAAI/bge-m3**（支援中英，效果好）、**GTE-base**、或 **Qwen3-Embedding** 小版本。
   - 避免太大 embedding 模型，否則向量資料庫和檢索會變慢。

3. **向量資料庫 + 檢索策略**：
   - 小規模資料用 **Chroma / FAISS** 即可；中大型用 **Milvus / Qdrant**。
   - 檢索時 **top_k = 4~8**（3B 模型 context 有限，不要超過 10）。
   - 加入 **重排序（reranker）**：如 bge-reranker，過濾掉不相關 chunk，顯著提升小模型表現。
   - 可考慮 **Hybrid Search**（向量 + 關鍵字/BM25），特別適合產品型號、報價單編號這種精準查詢。

4. **Prompt 工程（對小模型至關重要）**：
   - 明確指示：「只根據以下提供的知識卡片回答，不要自行推測。如果資訊不足，請說『根據目前資料無法確認』。」
   - 把 retrieved chunks 用清晰標籤分隔，例如：
     ```
     [知識卡片 1] 標題：...
     內容：...
     ---
     [知識卡片 2] ...
     ```

5. **進階減負 & 提升效益**：
   - **父子 Chunk（Parent-Child）**：小 chunk 用來檢索，大 chunk（父）用來生成。
   - **Metadata 過濾**：在向量資料庫中加入 metadata（如客戶類型、產品類別、更新日期），查詢時先過濾再向量搜尋，可大幅減少無關結果。
   - **定期更新機制**：報價、庫存、客戶狀態變化快，要設計增量更新，只更新變動的卡片。
   - **評估循環**：先做 50–100 張代表性卡片，測試常見問題，調整粒度與格式。

### 4. 推薦技術棧（本地友好）
- **框架**：LlamaIndex 或 LangChain（LlamaIndex 在知識庫管理上更成熟）。
- **本地運行**：Ollama + Llama 3.2 3B + 上述 embedding。
- **如果資料量大**：考慮混合結構化資料庫（產品/客戶用傳統 DB）+ RAG（文件/歷史報價）。

**總結建議優先順序**：
1. 先把**產品/物料**和**報價單**做成高品質的單一實體結構化卡片（這兩類查詢頻率最高）。
2. 客戶資料用「客戶為中心」聚合。
3. 嚴格控制 chunk 長度和單一性。
4. 多測試、多迭代（用真實業務問題驗證）。

這樣規劃，能讓 3B 小模型在資源有限的情況下，達到接近更大模型的實用效果，同時向量庫不會爆炸、檢索速度快、回答穩定。
